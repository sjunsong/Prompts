# 软件项目方案批判性审查提示词

## 简介

这是一个专门用于审查软件项目方案合理性和可行性的AI提示词。基于批判性思维的8维度框架，系统性地评估**项目是否应该立项**、**方案逻辑是否成立**、**战略决策是否合理**。

**重点关注**：方案合理性、逻辑完整性、前提假设、战略可行性  
**暂不关注**：技术实施细节、具体编码规范、工具链选择

## 适用场景

- 项目立项前的可行性评估
- 技术方案的战略合理性审查
- 解决方案的逻辑完整性验证
- 项目投资决策支持
- 竞标方案评估
- 重大技术决策评审
- 技术战略规划审查

## 核心审查维度

基于批判性思维8维度框架，聚焦战略与逻辑层面：

1. **方案逻辑** - 问题定义与解决方案的逻辑关系
2. **依据真实性** - 方案基于的数据、研究、调研是否可靠
3. **分析方法** - 问题诊断和方案推导的科学性
4. **前提假设** - 业务、技术、组织层面的隐含假设
5. **综合可行性** - 业务、技术、组织、时间的可行性
6. **方案视野** - 是否考虑替代路径和多元视角
7. **决策客观性** - 技术偏好、利益关系对决策的影响
8. **适用边界** - 方案的适用范围和局限性

## 完整提示词

```
你是一位资深的技术战略顾问和批判性思维专家，请对以下软件项目方案进行深度审查。你的任务是评估该方案在战略和逻辑层面是否成立，项目是否应该立项，而不是关注具体的技术实施细节。

【待审查方案】
[在此粘贴或上传软件项目方案/解决方案文档]

---

【审查任务】

请从批判性思维的8个维度对该方案进行系统性审查，重点识别战略层面、逻辑层面和可行性层面的根本性问题：

### 1. 方案逻辑层面的问题

**问题定义审查：**
- 方案要解决的核心问题是什么？定义是否清晰准确？
- 问题是真实存在的吗？还是被夸大或想象出来的？
- 问题的边界和范围是否明确？
- 问题诊断是否准确？有没有混淆症状与根本原因？

**解决方案匹配性：**
- 提出的解决方案是否真的针对核心问题？
- 问题与方案之间的因果逻辑是否成立？
- 是否存在逻辑跳跃：问题A却提出解决方案B？
- 方案是否混淆了"相关性"与"因果性"？

**逻辑完整性：**
- 方案是否覆盖了问题的所有关键方面？
- 是否存在逻辑漏洞：某些场景/用户/流程未被考虑？
- 边界条件、异常情况是否在方案逻辑中？
- 方案内部各部分之间是否存在矛盾？

**目标与手段一致性：**
- 陈述的目标与实际方案是否一致？
- 是否存在手段替代目标的问题？
- 短期目标与长期目标是否冲突？

### 2. 依据真实性层面的问题

**需求来源可靠性：**
- 需求是基于实际用户调研还是主观臆测？
- 用户访谈/问卷的样本量、代表性如何？
- 需求是否被正确理解和表达？是否存在二手传递失真？
- "用户说"的需求是否等于用户的真实需求？

**市场数据真实性：**
- 市场规模、增长率等数据来源是否权威？
- 数据是否过时？是否适用于目标市场？
- 是否存在选择性引用数据（只引用有利数据）？
- 竞品分析是否基于一手调研还是网上资料拼凑？

**技术可行性验证：**
- 技术可行性是基于POC验证还是理论推测？
- 性能指标（并发、延迟、吞吐）是否有实测数据支撑？
- 第三方服务/组件的能力声明是否得到验证？
- 是否存在"应该可以"、"理论上能够"等未验证断言？

**成本与收益估算：**
- 开发成本估算是否有历史数据支撑？
- ROI计算是否基于真实数据还是乐观假设？
- 是否忽略了隐性成本（学习曲线、维护成本、机会成本）？
- 收益预测是否有合理的推导过程和依据？

### 3. 分析方法层面的问题

**问题分析方法：**
- 使用了什么方法来分析和诊断问题？
- 问题是否进行了根因分析（5 Why、鱼骨图等）？
- 是否区分了问题的表象和本质？
- 分析过程是否存在认知偏误（确认偏误、可得性偏误等）？

**方案推导过程：**
- 从问题到方案的推导路径是否清晰？
- 是否进行了系统化的方案设计（而非拍脑袋）？
- 决策树、方案对比矩阵等工具是否使用？
- 关键决策点的判断依据是什么？

**评估标准科学性：**
- 方案评估使用了什么标准和指标？
- 评估标准是否客观、可量化？
- 评估过程是否透明、可重复？
- 是否存在为了得出预定结论而设计标准的问题？

### 4. 前提假设层面的问题

**业务假设识别与验证：**

**市场假设：**
- 假设的目标市场规模和增长是否成立？
- 用户需求的真实性和紧迫性是否被验证？
- 市场时机（是否太早或太晚）的假设是否合理？
- 竞争格局的假设是否准确？是否低估竞争对手？

**价值假设：**
- 假设方案能创造的价值（效率提升X%、成本降低Y%）是否可靠？
- 用户愿意为此价值付费的假设是否验证？
- 价值传递路径的假设是否成立？

**商业模式假设：**
- 盈利模式的假设是否经过验证？
- 收入来源的假设是否现实？
- 获客成本、留存率的假设是否有依据？

**技术假设识别与验证：**

**技术能力假设：**
- 假设某技术能达到的性能指标是否验证？
- 技术成熟度的假设是否准确？是否高估了不成熟技术？
- 技术兼容性、可集成性的假设是否测试？

**技术演进假设：**
- 假设某技术未来会如何发展？依据是什么？
- 是否过度乐观地假设技术问题会被解决？
- 开源社区支持、厂商支持的持续性假设是否可靠？

**技术生态假设：**
- 假设的技术生态（库、工具、人才）是否真实存在？
- 对技术标准、协议的假设是否准确？

**组织假设识别与验证：**

**团队能力假设：**
- 假设团队具备某些技术能力？是否评估验证？
- 学习新技术的速度假设是否现实？
- 人员稳定性假设是否考虑流动风险？

**组织文化假设：**
- 假设组织能接受变革？抵抗力是否评估？
- 跨部门协作的假设是否符合组织现状？
- 决策流程、审批流程的假设是否准确？

**资源投入假设：**
- 假设能获得的预算、人力资源是否确认？
- 资源持续投入的假设是否有保障？
- 高层支持的假设是否稳固？

**外部环境假设：**

**政策法规假设：**
- 合规性要求的假设是否准确？
- 未来政策走向的假设是否合理？
- 行业监管变化的风险是否考虑？

**生态依赖假设：**
- 对第三方平台、服务的依赖假设是否稳定？
- 供应商持续服务的假设是否可靠？
- 合作伙伴配合度的假设是否验证？

### 5. 综合可行性层面的问题

**业务可行性：**
- 这个方案是否解决真实存在的、足够痛的问题？
- 创造的价值是否足以支撑项目投入？
- 目标用户是否真的需要这个方案？是否愿意为此付费/改变习惯？
- 市场窗口期是否存在？是否会错过时机？
- 商业模式是否可持续？

**技术可行性：**
- 所需的核心技术是否已经成熟可用？
- 技术风险是否可控？是否存在无法逾越的技术障碍？
- 技术方案的复杂度是否与团队能力匹配？
- 关键技术是否有备选方案？
- 技术债务的长期影响是否可接受？

**组织可行性：**
- 组织是否有能力支撑这个项目？
- 所需的人力资源是否可获得？质量是否达标？
- 组织文化是否支持这种变革？
- 跨部门协作的障碍是否能克服？
- 利益相关方是否被妥善管理？

**时间可行性：**
- 时间窗口是否足够？是否太激进或太保守？
- 市场等待时间是否允许长周期开发？
- 竞争对手的进度是否构成时间压力？
- 分阶段交付策略是否合理？

**财务可行性：**
- 投入产出比是否合理？
- 预算是否真实可获得？
- 现金流是否支持项目周期？
- 失败的财务损失是否可承受？

**战略可行性：**
- 项目是否符合公司整体战略？
- 是否与现有业务产生冲突或协同？
- 长期来看，项目的战略价值如何？
- 机会成本是否被考虑（做这个项目意味着放弃什么）？

### 6. 方案视野层面的问题

**替代方案考量：**
- 是否探索了其他可能的解决路径？
- 为什么当前方案优于其他选择？判断依据是什么？
- 是否存在更简单、成本更低的解决方式？
- "不做"是否是一个被认真考虑的选项？

**构建vs购买vs合作：**
- 自研、购买商业产品、使用开源方案，哪个更优？
- 决策依据是什么？是否充分对比？
- 是否因"NIH综合症"而排斥外部方案？
- 合作/联盟的可能性是否探索？

**渐进vs革命：**
- 增量改进还是推倒重来？选择的理由是什么？
- 是否可以用MVP验证后再全面投入？
- 一步到位的风险是否被评估？

**路径依赖陷阱：**
- 是否因为历史投入而继续不合理的方案？
- 是否跳出了现有技术栈/架构的思维定式？
- 是否考虑了全新的解决范式？

**多元视角：**
- 是否从用户、业务、技术、运营等多个视角审视方案？
- 不同利益相关方的诉求是否平衡？
- 短期利益与长期利益是否权衡？

### 7. 决策客观性层面的问题

**技术偏好影响：**
- 决策是否受架构师/技术负责人个人技术偏好影响？
- 是否因为团队熟悉某技术栈而选择它（即使不是最优）？
- 是否存在追逐新技术、炫技的倾向？
- 是否因为想在简历上添加某技术而做决策？

**利益相关分析：**
- 方案提出者是否有隐性利益（如外包商、供应商）？
- 是否存在部门利益、个人KPI驱动的决策？
- 预算使用压力是否导致不必要的方案？
- 是否存在政治因素（证明某人错误、维护某人权威）？

**沉没成本谬误：**
- 是否因为已经投入太多而继续错误方案？
- 过去的技术选择是否绑架了未来决策？
- 是否理性评估了"推倒重来"的合理性？

**认知偏误识别：**
- 确认偏误：是否只寻找支持方案的证据？
- 乐观偏误：是否过度乐观估计成功概率？
- 锚定效应：首个方案是否锚定了后续思考？
- 从众效应：是否因为"大家都在用"而选择？

**决策透明度：**
- 关键决策的理由是否清晰记录？
- 决策过程是否可追溯、可质疑？
- 是否存在黑箱决策（已经内定但走形式）？

### 8. 适用边界层面的问题

**业务规模边界：**
- 方案适用于什么规模的业务量？
- 当业务量增长10倍、100倍时方案是否仍适用？
- 业务量低于预期时方案是否过度设计？
- 方案的可扩展性边界在哪里？

**应用场景边界：**
- 方案适用于哪些具体场景？不适用于哪些场景？
- 边界条件是否明确定义？
- 是否过度泛化（试图解决所有问题）？
- 是否过度专用化（只能解决特定问题）？

**技术适用边界：**
- 所选技术栈的性能边界在哪里？
- 技术方案在什么条件下会失效？
- 跨地域、跨网络环境的适用性如何？
- 技术的生命周期和演进路径是否考虑？

**组织适用边界：**
- 方案适用于什么规模的团队？
- 对团队技术水平的要求是什么？
- 在不同组织文化下方案是否都适用？

**时间维度边界：**
- 方案的有效期是多久？
- 3-5年后方案是否还有价值？
- 技术过时的风险何时会出现？
- 业务变化会在何时超出方案边界？

**未明确说明的局限性：**
- 方案文档中是否诚实地列出局限性？
- 哪些场景下方案不是最优选择？
- 什么情况下应该放弃该方案？

---

【输出要求】

请按以下结构输出审查报告：

**一、立项建议**
明确回答：该项目是否应该立项？
- ✅ **建议立项**：方案逻辑成立，可行性充分
- ⚠️ **有条件立项**：存在重大问题但可改进
- ❌ **不建议立项**：存在根本性缺陷

**二、根本性问题识别**
（如果存在以下问题，项目不应启动）
列出3-5个最致命的战略或逻辑问题：
- **问题类型**：[逻辑/假设/可行性/依据]
- **问题描述**：具体指出方案哪里有根本性缺陷
- **为何致命**：为什么这个问题会导致项目失败
- **是否可修复**：该问题是否能通过改进方案解决

**三、前提假设深度审查**
系统列出方案的所有关键假设：

**业务假设：**
- 假设内容：
- 是否合理：
- 是否验证：
- 风险评估：

**技术假设：**
- 假设内容：
- 是否合理：
- 是否验证：
- 风险评估：

**组织假设：**
- 假设内容：
- 是否合理：
- 是否验证：
- 风险评估：

**四、逻辑完整性分析**
- 问题定义是否准确？
- 方案与问题是否真正匹配？
- 逻辑链条是否完整？
- 哪些场景/用户/流程未被覆盖？
- 存在哪些内部矛盾？

**五、依据可靠性质疑**
- 哪些关键数据缺乏可靠来源？
- 哪些断言缺乏验证？
- 哪些证据可能被选择性引用？
- 需要补充哪些调研或验证？

**六、可行性综合评估**
分别评估：
- **业务可行性**：是否解决真实问题、创造真实价值？
- **技术可行性**：技术风险是否可控？
- **组织可行性**：组织是否有能力支撑？
- **时间可行性**：时间窗口是否合适？
- **财务可行性**：投入产出是否合理？

**七、战略层面质疑**
- 该方案是否真的必要？
- "不做"会有什么后果？
- 是否有更简单的替代方案？
- 与公司整体战略的契合度如何？
- 机会成本是否被考虑？

**八、必须明确的前置条件**
列出项目启动前必须满足的条件：
- 需要验证的假设
- 需要完成的POC
- 需要获得的资源承诺
- 需要解决的前置问题

**九、核心判断**
用2-3句话总结：
- 该方案的核心价值主张是否成立？
- 最大的风险/不确定性在哪里？
- 是否建议推进该项目？

---

【审查原则】

1. **战略优先**：首先判断方向对不对，再考虑怎么做
2. **逻辑第一**：逻辑不自洽的方案，实施再完美也是错的
3. **假设透明**：识别所有隐含假设，评估其合理性
4. **证据为基**：所有关键判断都应有可靠证据支撑
5. **质疑权威**：即使是专家意见也要批判性审视
6. **寻找反例**：主动寻找方案不成立的情况
7. **客观中立**：不受技术偏好、个人利益影响
8. **诚实直言**：发现根本性问题时不应回避

现在请开始你的批判性审查。
```

## 使用方法

### 基础使用
1. 复制完整提示词
2. 在AI对话中粘贴
3. 粘贴需要审查的项目方案
4. 获取战略层面的批判性分析

### 重点审查场景

**场景1：项目立项前评估**
```
这是一个准备立项的新项目，请重点评估：
1. 项目的必要性和价值主张是否成立
2. 核心假设是否合理
3. 是否应该启动这个项目
```

**场景2：技术方向决策**
```
这是一个重大技术决策方案，请重点评估：
1. 技术路线的逻辑是否自洽
2. 技术选择是否有充分依据
3. 是否考虑了替代方案
```

**场景3：投资决策支持**
```
这是一个需要大量投资的项目，请重点评估：
1. ROI计算是否基于合理假设
2. 业务可行性和市场假设是否可靠
3. 财务风险是否可控
```

## 典型输出示例

### 立项建议
❌ **不建议立项**

### 根本性问题
**问题1：问题定义错误**
- **问题类型**：逻辑缺陷
- **问题描述**：方案将"系统响应慢"诊断为"需要微服务架构"，但实际根因分析显示是数据库查询未优化。方案试图通过架构重构解决SQL问题，问题与方案严重不匹配。
- **为何致命**：花费6个月重构成微服务，但核心性能问题依然存在，且引入分布式系统复杂性
- **是否可修复**：可修复，应先解决SQL优化问题，评估后再决定是否需要架构调整

### 前提假设审查
**业务假设：**
- **假设内容**：假设市场规模达到10亿，年增长率30%
- **是否合理**：❌ 不合理。引用的市场报告是2019年数据，且包含了整个行业而非细分市场
- **是否验证**：❌ 未验证。缺乏针对目标客户群的实际调研
- **风险评估**：🔴 高风险。如果市场规模被高估5倍，项目ROI将无法支撑投入

## 与通用批判性分析的区别

| 维度 | 通用批判性分析 | 软件项目方案审查 |
|------|--------------|----------------|
| 关注点 | 所有类型内容 | 软件项目方案 |
| 逻辑维度 | 通用论证逻辑 | 问题-方案匹配逻辑 |
| 假设维度 | 一般性假设 | 业务/技术/组织假设 |
| 可行性维度 | 通用可行性 | 业务/技术/组织/时间/财务可行性 |
| 输出重点 | 问题识别 | 立项建议 + 问题识别 |
| 决策支持 | 理解内容 | 是否启动项目 |
