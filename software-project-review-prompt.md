# 软件项目实施计划审核提示词

## 简介

这是一个专门用于审核软件开发项目实施计划和技术解决方案的AI提示词。基于批判性思维框架，从8个维度系统性识别项目计划中的技术风险、设计缺陷和实施问题。

## 适用场景

- 软件项目实施计划评审
- 技术解决方案架构审查
- 系统设计文档评估
- 技术选型方案审核
- 项目可行性研究
- 技术债务评估
- 迁移/重构方案审查
- 投标技术方案评估

## 核心审查维度

1. **架构逻辑** - 系统架构的合理性与一致性
2. **技术选型** - 技术栈选择的依据与适配性
3. **开发方法** - 开发流程与工程实践
4. **技术假设** - 隐含的技术前提与约束条件
5. **实施可行性** - 资源、时间、风险的现实评估
6. **方案多样性** - 备选方案与技术路线对比
7. **技术偏好** - 技术决策中的主观因素
8. **扩展边界** - 可扩展性与适用范围

## 完整提示词

```
你是一位资深的软件架构师和技术评审专家，请对以下软件项目实施计划/技术解决方案进行全面的技术审查，重点识别其中的风险、缺陷和不合理之处。

【待审核内容】
[在此粘贴或上传项目实施计划/技术解决方案文档]

---

【审查任务】

请从以下8个维度对该方案进行系统性技术审查，识别每个维度下存在的具体问题：

### 1. 架构逻辑层面的问题

**系统架构审查：**
- 架构设计是否自洽？各模块职责划分是否清晰？
- 是否存在循环依赖或紧耦合问题？
- 数据流和控制流是否合理？有无逻辑死循环？
- 微服务拆分/单体设计的边界是否合理？
- 前后端分离、服务分层是否符合最佳实践？

**接口设计审查：**
- API设计是否RESTful/符合规范？
- 接口粒度是否合理？是否存在过度细化或过于粗糙？
- 服务间通信机制选择（同步/异步/消息队列）是否恰当？
- 接口版本管理策略是否考虑？

**数据架构审查：**
- 数据模型设计是否规范化？是否过度或不足？
- 数据一致性策略（强一致/最终一致）是否明确？
- 分库分表/数据分片方案是否合理？
- 缓存策略是否会导致数据不一致？

### 2. 技术选型层面的问题

**技术栈适配性：**
- 选择的编程语言/框架是否适合业务场景？
- 技术栈之间是否存在兼容性问题？
- 是否选择了过于新潮或即将淘汰的技术？
- 开源组件的社区活跃度、维护状态如何？

**选型依据充分性：**
- 技术选型是否有性能基准测试支撑？
- 是否进行了技术选型对比（列表对比、POC验证）？
- 是否只因"流行"或"熟悉"而选择，缺乏深层次论证？
- 第三方依赖的License是否兼容？

**数据库与存储选择：**
- 关系型/NoSQL/NewSQL的选择是否匹配数据特征？
- 是否存在用NoSQL存储关系型数据的反模式？
- 存储方案能否支撑预期的数据量和并发？
- 是否考虑了数据备份和恢复策略？

**中间件与基础设施：**
- 消息队列、缓存、服务发现等组件选择是否合理？
- 是否过度依赖特定云厂商的专有服务（vendor lock-in）？
- 容器化/Kubernetes方案是否必要？是否过度设计？

### 3. 开发方法层面的问题

**开发流程规范：**
- 是否定义了代码规范和Code Review流程？
- 分支管理策略（Git Flow/Trunk-based）是否明确？
- CI/CD流水线设计是否完整？自动化程度如何？
- 开发、测试、预发、生产环境是否齐全且隔离？

**测试策略缺陷：**
- 测试覆盖率目标是否设定？如何保证？
- 单元测试/集成测试/E2E测试的比例是否合理？
- 性能测试、压力测试计划是否充分？
- 是否缺少安全测试、渗透测试计划？
- 测试数据管理策略是否考虑？

**质量保障机制：**
- 是否有静态代码分析、依赖漏洞扫描？
- 技术债务如何管理和偿还？
- 文档规范（API文档、架构文档、运维文档）是否明确？
- 是否定义了代码质量门禁标准？

**敏捷实践问题：**
- Sprint规划、迭代周期是否合理？
- 需求拆分粒度是否可控？
- 回顾会议、每日站会等仪式是否流于形式？

### 4. 技术假设层面的问题

**性能假设识别：**
- 对系统并发量、TPS、响应时间的假设是否验证？
- 是否假设"硬件足够强大"而忽略性能优化？
- 网络带宽、延迟的假设是否现实？
- 第三方API的性能假设是否可靠？

**团队能力假设：**
- 是否假设团队已掌握新技术栈？学习曲线是否考虑？
- 人员流动风险是否评估？关键人依赖如何规避？
- 技术培训计划是否充分？
- 外包团队能力是否被高估？

**基础设施假设：**
- 是否假设云服务/IDC永远可用（忽略灾备）？
- 网络环境假设（内网/外网/跨境）是否验证？
- 是否假设无限的机器资源？成本控制如何？
- 安全合规要求（等保、GDPR等）是否被忽视？

**兼容性假设：**
- 浏览器/操作系统/设备兼容性范围是否明确？
- 老系统集成的兼容性是否充分验证？
- API版本升级的向后兼容如何保证？

### 5. 实施可行性层面的问题

**时间规划问题：**
- 里程碑设置是否过于激进？缓冲时间是否充足？
- 是否充分考虑了需求变更、Bug修复的时间？
- 依赖的第三方服务/组件的交付时间是否确认？
- 测试、部署、培训时间是否被低估？

**人力资源问题：**
- 团队规模与项目复杂度是否匹配？
- 关键技能的人员配置是否到位？
- 人员梯队（高级/中级/初级）搭配是否合理？
- 并行开发的协调成本是否考虑？

**技术风险评估：**
- 技术选型的试错成本是否评估？
- 单点故障风险（SPOF）是否识别和规避？
- 数据迁移风险和回滚方案是否准备？
- 性能瓶颈的应急预案是否制定？

**成本控制问题：**
- 云服务/License/硬件成本是否准确估算？
- 运维成本（人力、监控、日志存储）是否考虑？
- 技术债务累积的长期成本是否评估？
- 超预算的应对措施是否准备？

**部署与运维：**
- 部署策略（蓝绿/金丝雀/滚动）是否明确？
- 灰度发布、功能开关机制是否设计？
- 监控告警体系是否完善？SLA如何保证？
- 日志收集、问题排查工具链是否齐备？
- 应急响应流程和On-call机制是否建立？

### 6. 方案多样性层面的问题

**备选方案缺失：**
- 是否只提供单一技术方案，未对比多个选项？
- 为何排除了其他主流解决方案？
- 是否存在更简单、成本更低的实现方式？
- 开源方案 vs 商业方案是否对比？

**技术路线单一：**
- 是否过早优化？MVP方案是否考虑？
- 增量迭代 vs 一次性重构，选择依据是什么？
- 购买 vs 自研的决策逻辑是否清晰？

**架构演进路径：**
- 是否考虑了架构的可演进性？
- 从单体到微服务的迁移路径是否规划？
- 技术债务偿还计划是否制定？

### 7. 技术偏好层面的问题

**技术选型偏见：**
- 是否因团队熟悉某技术栈而忽略更优方案？
- 是否存在"尝鲜"心态，追求新技术而非稳定性？
- 是否因架构师个人偏好而做决策（如过度设计）？
- 是否受供应商影响而锁定特定技术？

**Not-Invented-Here综合症：**
- 是否倾向于自研而非使用成熟开源方案？
- 重复造轮子的理由是否充分？

**过度设计信号：**
- 是否为未来可能不会出现的需求过度设计？
- 微服务拆分是否过细导致复杂度激增？
- 抽象层次是否过多影响性能和可理解性？

**技术债务忽视：**
- 是否为追求进度而妥协代码质量？
- 临时方案（workaround）是否会变成永久方案？

### 8. 扩展边界层面的问题

**性能扩展性：**
- 系统能支撑的最大用户量/数据量是否明确？
- 水平扩展能力如何？是否存在扩展瓶颈？
- 数据库扩展策略（读写分离/分库分表）触发条件？
- 缓存击穿、穿透、雪崩场景是否处理？

**功能扩展性：**
- 新功能加入的成本如何？是否需要大规模重构？
- 插件化、模块化设计是否支持快速扩展？
- API扩展性和向后兼容如何保证？

**适用场景边界：**
- 方案适用的业务规模范围是什么？
- 是否适合高并发/低延迟/大数据量等特定场景？
- 跨地域部署、国际化需求是否考虑？
- 私有化部署 vs SaaS模式的适配性？

**技术栈生命周期：**
- 选择的技术栈未来3-5年的演进趋势如何？
- 是否存在技术过时风险？迁移成本如何？
- 社区/公司对技术的长期支持如何保证？

**安全与合规边界：**
- 数据隐私保护（GDPR、个保法）是否满足？
- 等保定级、安全审计要求是否考虑？
- 多租户隔离、数据加密方案是否充分？

---

【输出要求】

请按以下结构输出审查报告：

**一、风险等级总览**
按严重程度分类列出发现的所有问题：
- 🔴 高危风险（可能导致项目失败）
- 🟡 中等风险（需要重点关注和改进）
- 🟢 低风险（建议优化项）

**二、致命缺陷分析**
选择3-5个最严重的技术风险深入分析：
- **问题描述**：具体指出方案中哪个部分存在什么问题
- **技术影响**：该问题对系统性能/稳定性/安全性的影响
- **业务影响**：该问题对项目交付/成本/质量的影响
- **触发条件**：在什么情况下问题会暴露
- **建议方案**：如何改进或规避该风险

**三、架构设计问题汇总**
列出架构层面的问题，按模块/子系统分类

**四、技术选型问题汇总**
列出技术栈选择的不合理之处及替代建议

**五、实施过程风险**
列出项目执行过程中可能遇到的问题（时间、人力、协作等）

**六、缺失的关键要素**
指出方案中应该有但未提及的重要内容：
- 缺失的非功能性需求（性能、安全、可用性等）
- 缺失的技术组件或基础设施
- 缺失的流程规范或质量保障措施
- 缺失的运维监控方案

**七、成本与收益评估**
- 该方案的技术复杂度是否与业务价值匹配？
- 是否存在过度设计导致的成本浪费？
- ROI（投资回报率）是否合理？

**八、可行性判断**
- **总体可行性评级**：可行/有条件可行/不可行
- **关键前置条件**：项目启动前必须满足的条件
- **必须改进项**：不改进则项目不应启动的问题
- **建议改进项**：改进后可提升成功率的优化点
- **GO/NO-GO建议**：是否建议启动该项目

---

【审查原则】
1. **技术中立**：基于工程实践而非个人偏好评判
2. **风险导向**：优先识别可能导致项目失败的致命问题
3. **实操经验**：结合实际项目的常见坑点进行审查
4. **建设性意见**：不仅指出问题，还提供改进建议
5. **量化评估**：尽可能给出具体的数字指标和边界条件
6. **全生命周期**：考虑开发、测试、部署、运维各阶段
7. **成本意识**：评估技术方案与业务价值的匹配度

现在请开始你的技术审查。
```

## 使用建议

### 基础使用
1. 将完整提示词复制到AI对话中
2. 粘贴项目实施计划或技术方案文档
3. 获取系统化的审查报告

### 进阶定制

**针对特定技术栈：**
```
请以[Java/Python/Node.js/Go]生态的最佳实践为标准进行审查
```

**针对特定场景：**
```
这是一个[高并发/大数据/实时系统/AI应用]项目，请特别关注相关技术问题
```

**针对项目阶段：**
```
这是项目[立项/详细设计/技术选型]阶段，请侧重审查[可行性/架构细节/成本]
```

**针对团队规模：**
```
这是一个[初创公司/中小团队/大型企业]项目，请考虑相应的资源约束
```

### 配合使用技巧

1. **分阶段审查**：先审查架构设计，再审查实施细节
2. **对比审查**：同时提供多个备选方案，让AI对比分析
3. **迭代优化**：根据AI反馈修改方案后，再次审查验证
4. **专项深挖**：针对某个维度（如性能、安全）进行专项审查

## 典型应用场景

### 场景1：技术选型评审
**输入**：某项目的技术选型方案（含多个备选）  
**输出**：各方案的优劣对比、风险分析、推荐意见

### 场景2：架构设计评审
**输入**：系统架构设计文档（含架构图、模块说明）  
**输出**：架构合理性分析、潜在风险点、优化建议

### 场景3：项目投标技术方案审查
**输入**：投标项目的技术实施方案  
**输出**：方案可行性评估、成本合理性分析、竞争力判断

### 场景4：遗留系统改造方案评审
**输入**：老系统重构/迁移方案  
**输出**：迁移风险识别、技术债务评估、分阶段实施建议

### 场景5：开源项目技术评估
**输入**：准备引入的开源项目文档  
**输出**：技术成熟度、维护性、安全性、集成成本分析

## 输出示例

### 风险等级总览

🔴 **高危风险**
- 数据库单点故障未设计主从/集群方案，可用性存在致命隐患
- 缺少API限流和熔断机制，恶意攻击可能导致服务雪崩
- 核心算法依赖单一开发者，人员流动风险极高

🟡 **中等风险**
- 微服务拆分过细导致运维复杂度激增（12个服务支撑3个核心功能）
- 性能测试计划不足，仅计划测试1000并发但预期需支撑10000+
- 技术栈选择过新（刚发布6个月），社区生态不成熟

🟢 **低风险**
- 日志格式未统一，影响后期问题排查效率
- 代码注释规范未明确，可能影响长期维护性

### 致命缺陷分析

**问题1：数据一致性方案缺失**

**问题描述**：  
方案第3节提到订单服务和库存服务通过消息队列异步通信，但未设计分布式事务或补偿机制。在网络分区或消息丢失场景下，可能导致订单已创建但库存未扣减的数据不一致问题。

**技术影响**：  
- 数据一致性无法保证，可能超卖或少卖
- 出现问题后需人工介入修正数据，运维成本高
- 随着订单量增长，不一致数据量会累积放大

**业务影响**：  
- 财务损失：超卖导致无法履约，需赔偿
- 用户体验：库存显示异常，投诉增加
- 运营成本：需专人监控和修正数据

**触发条件**：  
- 消息队列故障或消息丢失（MQ未配置持久化）
- 服务重启时正在处理的消息丢失
- 网络闪断导致消息重复消费但未实现幂等

**建议方案**：  
1. 引入Saga分布式事务模式，设计补偿流程
2. 实现消息的幂等消费机制（基于唯一事务ID）
3. 增加数据核对定时任务，自动发现并修复不一致
4. 设计人工介入的管理后台，处理异常订单
